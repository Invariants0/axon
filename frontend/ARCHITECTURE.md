# AXON Frontend Architecture

This document describes the architecture of the AXON frontend application.

## Core Principles

1.  **Observability:** The UI is designed as an "observatory" to watch the AI system evolve. It prioritizes real-time feedback, clear logs, and visual representations of the system's state.
2.  **Modularity:** The application is built using reusable components and a clear separation of concerns (UI, state, data fetching, real-time communication).
3.  **Performance:** The UI must remain responsive even when handling a high volume of logs or complex graph visualizations.

## Key Technologies

*   **Next.js (App Router):** Provides the foundation for routing, server-side rendering (where applicable), and API routes.
*   **Tailwind CSS & shadcn/ui:** Used for rapid, consistent, and accessible styling.
*   **Zustand:** A small, fast, and scalable bearbones state-management solution. Used for global application state (e.g., UI preferences, active persona, current evolution state).
*   **React Query:** Used for fetching, caching, synchronizing and updating server state (e.g., historical logs, system metrics).
*   **Framer Motion:** Used for smooth, declarative animations that enhance the user experience without being overwhelming.
*   **React Flow:** Used for rendering the interactive "Capability Graph" that visualizes the system's skills and their relationships.
*   **Monaco Editor:** Used for the "Code Evolution Viewer" to display the code generated by the AI system.

## Directory Structure

*   `app/`: Contains the Next.js App Router pages and layouts.
    *   `page.tsx`: The landing page.
    *   `dashboard/page.tsx`: The main control room dashboard.
    *   `layout.tsx`: The root layout, including global providers and fonts.
*   `components/`: Contains reusable UI components.
    *   `ui/`: shadcn/ui components.
    *   `axon/`: Custom components specific to the AXON application (e.g., LogViewer, CapabilityGraph, CodeViewer).
*   `hooks/`: Contains custom React hooks.
    *   `use-mock-websocket.ts`: A hook that simulates a WebSocket connection for development and demonstration purposes.
*   `lib/`: Contains utility functions and mock data.
    *   `utils.ts`: General utility functions (e.g., `cn` for Tailwind class merging).
    *   `mock-data.ts`: Mock data for logs, graph nodes, and edges.
*   `store/`: Contains Zustand store definitions.
    *   `app-store.ts`: The main application store.
*   `styles/`: Contains global CSS and Tailwind configuration.
    *   `globals.css`: Global styles and Tailwind directives.

## State Management Strategy

*   **Local State:** Used for component-specific state (e.g., whether a dropdown is open, the current tab in a view).
*   **Global State (Zustand):** Used for state that needs to be accessed by multiple components across the application (e.g., the currently selected persona, whether the evolution sequence is active).
*   **Server State (React Query):** Used for data fetched from the backend API (e.g., historical logs, system metrics). React Query handles caching, background updates, and error handling.
*   **Real-time State (WebSockets):** Used for data that changes rapidly and needs to be pushed from the server to the client (e.g., live logs, graph updates). This state is typically managed within a custom hook (e.g., `useWebSocket`) and integrated with the global or local state as needed.

## Real-time Communication

The application relies heavily on real-time communication to provide a live view of the AI system's evolution. This is implemented using WebSockets.

*   A custom hook (e.g., `useWebSocket`) manages the WebSocket connection.
*   Incoming messages are parsed and dispatched to the appropriate state management solution (e.g., Zustand for global state updates, React Query for cache invalidation).
*   The UI components subscribe to the relevant state and re-render automatically when new data arrives.

## Theming and Styling

*   The application uses a dark, futuristic theme by default.
*   Tailwind CSS is used for all styling, providing a utility-first approach that ensures consistency and maintainability.
*   shadcn/ui provides a set of accessible, customizable components that serve as the building blocks for the UI.
*   CSS variables are used for defining the color palette, allowing for easy theming and customization.

## Animations

*   Framer Motion is used for all animations.
*   Animations are used purposefully to guide the user's attention, provide feedback on interactions, and enhance the overall "cinematic" feel of the application.
*   Key animation areas include page transitions, log entries appearing, graph nodes popping in, and the evolution pipeline visualization.
